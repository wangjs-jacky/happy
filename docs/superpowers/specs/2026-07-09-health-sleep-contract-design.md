# 健康打卡 · 睡眠数据契约打磨（最小闭环）— 设计文档

> 日期：2026-07-09 ｜ 分支：`health-sleep-contract`
> 一句话：把健康打卡「睡眠」这一类的数据契约固稳（字段名统一 + 格式统一 + 写完自校验），并把 Happy 右面板从「只有评分趋势」升级为「睡眠 Hero 大卡」，先跑通最小闭环，验证有效再扩到运动/饮食/体征。

## 一、背景与目标

`人生辅助系统/健康打卡/` 是一个「文件夹即 Agent」：用户丢图，健康打卡专家（`健康打卡/CLAUDE.md`）看图归类、把结构化数据写进 `日报/YYYY-MM-DD.md` 的 YAML frontmatter。Happy App 进入该会话时，右滑面板换成健康打卡专属面板（`HealthCheckinPanel`），实时读日报展示。

**问题（有真实数据佐证）**：数据契约会漂移，只有 `评分` 字段稳（因为它有强约束 + 面板依赖）。

- 「睡了多久」三种叫法：`2026-06-25` 用 `夜间睡眠/总睡眠/零星小睡`，`2026-07-06` 用 `总时长`，schema 里叫 `总时长`。
- schema 外即兴字段：`自拍`（顶层）、`超过用户`、`快速眼动`(REM)。
- `消耗卡路里: null` 违反「缺失整段省略」铁律。
- 面板信息密度 ≪ 数据密度：日报采了睡眠时长/深浅睡/REM/入睡起床，面板只展示「今日三对勾 + 评分趋势」。

**目标（本轮范围：只睡眠）**：
1. 睡眠字段名统一、格式统一，agent 写完能自校验。
2. 面板把睡眠时长/结构可视化（方向 B：睡眠 Hero 大卡）。
3. 跑通「字段表 + 格式 + 自检 + 面板」最小闭环，作为可复制的 Agent 范本方法论；运动/饮食/体征下一轮再套。

**非目标（YAGNI）**：运动/饮食/体征字段这轮不动；不写独立 validator 脚本（纯 CLAUDE.md 自检清单）；不做真 hypnogram（逐段时序），结构图只表达占比。

## 二、值格式策略（已定：C 统一带单位字符串）

frontmatter 的值**保留人类可读的带单位字符串**（如 `总时长: 7h20m`），不改成纯数字。「稳」的重心因此落在三点：**字段名统一 + 值格式统一 + 解析集中一处**。面板侧把带单位字符串→数值的解析收敛到 `healthLog.ts` 一个模块（`parseDuration`），只写一次。

## 三、数据契约（Obsidian 侧 · 改 `健康打卡/CLAUDE.md` 第三节睡眠部分）

### 3.1 睡眠字段权威表 v2（唯一合法字段名）

| 字段 | 格式 | 必写 | 说明 |
|------|------|:---:|------|
| `总时长` | `XhYm` | ★ | 当晚**实际睡眠**（夜间主睡，**不含**日间小睡）。面板趋势/复盘主指标 |
| `深睡` | `XhYm` | 可选 | |
| `浅睡` | `XhYm` | 可选 | |
| `快速眼动` | `XhYm` | 可选 | REM，正式字段（有就写） |
| `日间小睡` | `XhYm` | 可选 | 原「零星小睡」归此，不并进总时长 |
| `评分` | 纯数字 0-100 | ★ | 面板唯一强依赖，禁止带「分/星/%」 |
| `质量` | 文字等级 | 可选 | 差/一般/良好/优秀，禁止塞数字 |
| `超过用户` | `NN%` | 可选 | |
| `入睡` / `起床` | `"HH:MM"` | 可选 | 带引号防 YAML 误解析为时间 |
| `来源` | 文字 | 可选 | 设备/App，如「华为运动健康」 |
| `原图` | wikilink | 有图必写 | `"[[附件/YYYY-MM-DD/睡眠-1.png]]"` |

### 3.2 值格式规范

- 时长类（总时长/深睡/浅睡/快速眼动/日间小睡）：一律 `XhYm`，不足 1 小时也写 `0h55m`（避免 `55m`/`55min` 分叉）。
- 评分：纯数字，无单位、无「分/星」。
- 时间点：`"HH:MM"` 24 小时制、带引号。
- 百分比：`NN%`。
- **缺失整行省略，禁止 `null`/空占位/编造**。

### 3.3 关键约定

- `总时长` = 夜间主睡、不含小睡；需要「含小睡总量」时由聚合端 `总时长 + 日间小睡` 算，**不存冗余字段**。
- `快速眼动`(REM) 转正为可选字段。

### 3.4 写完自检清单（纯 CLAUDE.md 承载，agent 写完睡眠段逐条过，无脚本）

1. 时长字段叫 `总时长`（不是 夜间睡眠/总睡眠）？
2. 每个时长都是 `XhYm` 格式？
3. `评分` 是纯数字，没混进「分/星/超过 X%」？
4. `质量` 只有文字等级？
5. 没有 `null`/空占位？没有权威表外的即兴键？
6. `入睡`/`起床` 带引号？

## 四、历史迁移（只 2 篇有睡眠数据）

- `2026-06-25`：`夜间睡眠`→`总时长`；`零星小睡`→`日间小睡`；删冗余 `总睡眠`（= 总时长 + 日间小睡）；`超过用户 99%` 保留；`评分 89` 不动。
- `2026-07-06`：已基本合规，`快速眼动` 保留即可。
- `2026-06-17`/`06-23`/`06-24`：无睡眠段，不动。

## 五、Happy 面板（happy-app 侧 · 方向 B「睡眠 Hero 大卡」）

### 5.1 视觉结构（自上而下）

1. **睡眠 Hero 卡**（`expo-linear-gradient` 渐变，睡眠夜晚靛蓝，明暗主题各一套色）：
   - 左：`总时长` 大字 + 「今晚总睡眠」小字。
   - 右：**SVG 评分环**（`react-native-svg`，环占比 = 评分/100，颜色随评分档：绿/琥珀/红）。
   - 中：**睡眠结构双视图**——`堆叠条 ⇄ 甜甜圈`，卡内切换。
   - 底：一行三数字（深睡/浅睡/REM 占比）+ 入睡→起床。
   - ⚠️ 结构条/环表达**占比**（数据只有各阶段总时长，非逐段时序）；两端时间是入睡/起床。
2. **本周趋势卡**：`时长 / 评分` 切换 tab 的柱状图。
3. **今日打卡卡**：现有三对勾（运动/睡眠/饮食），下移到底部。
4. **记录按钮**：不变。

### 5.2 交互与偏好持久化

- 结构双视图偏好（`bar`|`donut`）与趋势 metric（`duration`|`score`）都用 `useLocalSetting` 持久化，下次打开保持上次选择。
- 默认：结构 `堆叠条`、趋势 `时长`。

### 5.3 `healthLog.ts` 改动（解析集中一处）

- `HealthLog` 类型扩容：新增 `总时长/深睡/浅睡/快速眼动/日间小睡`（分钟数值）、`入睡/起床`（字符串）、`质量`（字符串）等。
- 新增 `parseDuration('7h20m') → 440`（分钟）纯函数；`0h55m`/`7h20m` 都能解析，非法返回 `null`。
- 保留现有 `评分` 解析（`extractSleepScore`），契约收紧后 `质量` 兜底分支可留作历史容错。
- 面板结构占比 = 各阶段分钟 / 各阶段分钟之和（不用总时长做分母，避免清醒/未记录段导致不足 100%）。

## 六、组件边界与数据流

```
日报 *.md (YAML) ──sessionListDirectory/ReadFile RPC──▶ HealthCheckinPanel
                                                          │
                                    healthLog.ts (唯一解析层)
                                    · parseHealthLog(frontmatter) → HealthLog
                                    · parseDuration(str) → 分钟
                                                          │
                        ┌─────────────┬──────────────────┴────────┐
                     SleepHeroCard   SleepTrendCard            TodayChips
                     (评分环+结构双视图)  (时长/评分 tab)          (三对勾)
```

- 单一解析真相：所有「带单位字符串 → 数值」只在 `healthLog.ts` 发生，面板组件只消费 `HealthLog`。
- 契约真相：`健康打卡/CLAUDE.md` 睡眠字段表；`healthLog.ts` 按它实现。两端遵循同一份规范（人工保持一致，本轮不引入机器强约束）。

## 七、测试

- `parseDuration` 单测：`'7h20m'→440`、`'0h55m'→55`、`'8h0m'→480`、非法 `'abc'→null`、空→null。
- `parseHealthLog` 单测：喂迁移后的 `2026-06-25`/`2026-07-06` frontmatter，断言字段解析正确、评分趋势不回归。
- `pnpm typecheck` 通过。

## 九、风险与未决

- **明暗主题渐变**：Hero 卡渐变要在 light/dark 都好看，用半透明叠加适配 `theme.colors`，避免硬编码深色。
- **SVG 评分环成本**：`react-native-svg` 已在依赖，风险低；若环实现受阻，退化为评分胶囊徽章（不阻塞其余改动）。
- **占比 vs 时序**：本轮明确只做占比；若日后要真 hypnogram，需扩数据契约（agent 逐段读截图，多数手环无此数据），另立方案。
- **两端契约人工同步**：CLAUDE.md 与 `healthLog.ts` 靠人保持一致；若后续漂移复现，再考虑独立 schema 文件 / validator（本轮 YAGNI）。
