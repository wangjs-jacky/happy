### 2.3 Unistyles

## 三、发现的问题

- `WEB-069`：日志页的清除操作可以聚焦，但浏览器无法稳定解析其可访问名称。
- `WEB-070`：日志页外壳、日志表面、条目和空态使用固定浅色，在深色应用中形成主题断层。
- `WEB-071`：应用内测试页的汇总、结果和运行状态使用固定浅色，深色主题下对比与层级异常。
- `WEB-072`：Unistyles 的浅色、深色选项是可聚焦 `div`，没有单选组、单选项与选中状态。
- `WEB-073`：运行时详情开关缺少持久名称和可读取的选中状态。
- `WEB-074`：“切换配色方案”表现为按钮，但点击后只写普通日志，页面状态和反馈均不变化。
- `WEB-075`：从 Terminal 深色切到浅色再切回深色会落到 Caramel，无法保留当前主题包。
- `WEB-076`：多个章节标题和说明使用固定近黑文字，在深色主题中对比度不足。
- `WEB-077`：800px 视口下断点说明仍显示初始化时的 1440px，与页面实时宽度不一致。
- `WEB-078`：章节阴影无条件下发原生 `shadow*` 属性；独立审查确认 Web 会产生弃用 warning 的风险。

修复前所有安全动作新增 Console warning / error 为 0/0，失败 XHR/Fetch 为 0；问题主要来自语义、
无效果交互、主题和实时视觉状态，而不是请求失败。

- Unistyles 页在深色主题下出现低对比文字，800px 断点说明错误显示 1440px。

## 五、根因与修复

本批有五类根因：演示页沿用固定浅色；交互组件没有表达真实语义；无效果动作仍进入焦点序列；
主题切换只处理明暗、不保留主题包；模块初始化宽度与 Web 阴影属性没有区分运行时和平台。

本批修复：

1. 日志、测试和 Unistyles 页的外壳、表面、主次文字、结果行与运行状态接入当前主题。
2. 为日志操作补齐稳定按钮名称和测试标识，同时保持日志正文不进入审计证据。
3. 为测试汇总的总数、通过数和失败数增加稳定标识；回归断言改为总数大于零、总数等于通过数、失败数为零。
4. 将浅色、深色主题组织为具名 `radiogroup` 和两个互斥 `radio`，暴露实时选中状态。
5. 为运行时详情开关增加可访问名称、`checked` 状态和稳定测试标识。
6. 删除没有可见效果的配色方案按钮，使空操作不再进入按钮树和 Tab 顺序。
7. 新增主题模式解析函数：保留当前主题包，仅切换明暗；空值、未知值回退 Caramel。
8. 使用 `useWindowDimensions` 读取实时视口宽度，替代模块加载时固定读取。
9. Web 章节只使用 `boxShadow`；原生平台继续保留 `shadow*` 和 `elevation`。

### 8.2 可避免耗时与改进

1. **链式命令没有保持预期工作目录**
   - 现象：创建 worktree 后的首个依赖安装误在主仓库执行；没有产生 Git 修改，随后在 Batch 20 worktree 正确重跑。
   - 改进：创建、安装、启动分别使用显式 `workdir` 的独立命令，不依赖前一段命令的目录状态。
2. **视觉隐私边界应在截图前判定**
   - 现象：日志页包含用户运行时内容，不能像普通演示页一样截图。
   - 改进：候选页登记时先标记敏感正文；此类页面只检查容器样式、可访问树和计数，不生成截图。
3. **演示按钮必须验证业务效果**
   - 现象：首轮只确认配色方案按钮可点击，没有把“点击前后状态相同”立即判为失败。
   - 改进：所有可点击控件都必须有“操作前状态 → 操作 → 可观察结果”断言；没有结果的演示操作应移出焦点序列。
4. **固定测试总数会制造无关失败**
   - 现象：应用内套件增删时，硬编码 `50` 会让 UI E2E 无关失败。
   - 改进：断言不变量 `total > 0`、`passed === total`、`failed === 0`，并通过稳定标识读取三个值。
5. **最终回归顺序**
   - 本批先完成定向 GREEN、交叉审查和 Browser 复验，再运行一次完整回归，避免审查修订后重复长测试。
6. **Web Switch 的测试断言误用了 DOM 属性**
   - 现象：首轮完整 E2E 为 39/40；真实 DOM 使用原生 checkbox `checked` 状态，
     但测试要求一定存在 `aria-checked` 字符串属性。
   - 浪费：一次约 4.9 分钟的完整 E2E，以及修订后的全量重跑。
   - 改进：对 Web `switch` 使用 Playwright `toBeChecked()`，同时保留角色和可见名称定位；
     不从 React 属性推断最终 DOM 一定使用同名 ARIA 属性。

### 8.3 下批标准 SOP

1. 创建同级 worktree，后续每条命令显式指定 worktree。
2. 本地临时关闭 Watchman，启动 Web；提交前逐字节恢复。
3. 初始化 Browser Control 账本，按“基线 → 动作 → 可观察结果 → Console → Network → 安全截图”执行。
4. 页面含日志、会话或设备数据时，先定义字段与截图隐私边界。
5. 任何按钮都必须断言可见结果；无真实效果的控件从交互树移除。
6. 先完成定向 TDD、Browser 复验和独立审查，再执行一次最终完整 E2E。
7. 串行执行完整 Vitest、Web export，避免与 CPU 密集型 E2E 并发。
8. 恢复主题、开发者模式、视口、临时配置和原页面，校验账本后再提交 PR。
