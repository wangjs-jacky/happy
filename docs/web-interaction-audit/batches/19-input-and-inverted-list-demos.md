## 三、发现的问题

- `WEB-061`：LegendList 在 Web 运行时产生稳定 Console error，消息发送后不能留在列表中；其调试连接也失败。
- `WEB-062`：倒序列表的六个互斥选择和发送操作缺少与行为一致的角色、选中或禁用状态。
- `WEB-063`：倒序列表页面使用固定白色外壳，在深色应用中形成明显主题断层。
- `WEB-064`：五个多段文本输入只依赖 placeholder，没有与可见标题关联的持久名称。
- `WEB-065`：多段文本输入页使用固定白色和彩色浅背景，在深色应用中形成主题断层。
- `WEB-066`：22 张输入样式卡片可点击但没有单选组、单选项和唯一选中状态。
- `WEB-067`：22 个只读输入和 138 个静态装饰节点进入 Tab 顺序，用户会在不可编辑预览中反复停留。
- `WEB-068`：输入样式页的标题、说明、卡片和固定预览外壳使用固定浅色；示例内部品牌配色本身是演示内容，不应被统一改写。

输入样式页底部固定预览不是遮挡问题：在 800px、最大滚动位置下，最后一张卡片底部与固定预览顶部仍保留 36px 间距。

## 五、根因与修复

本批有四类根因：原生列表库被直接暴露给 Web；可交互演示控件没有表达真实语义；
演示页面使用固定浅色；静态预览复用了真实输入和操作组件，导致焦点树被装饰内容污染。

本批修复：

1. Web 禁用 LegendList 入口，并在渲染分支二次防守；原生平台继续保留 LegendList 演示。
2. 倒序列表两组选择组织为有名称的 `radiogroup`，六项使用 `radio` 角色和选中状态；发送按钮表达空输入禁用状态。
3. 三个页面的外壳、主次文字、表面、边框与输入颜色接入当前主题。
4. 五个多段文本输入使用对应可见标题作为稳定名称，并同时为通用实现和 Web 独立实现补齐属性转发。
5. 22 张输入样式卡组织为有名称的单选组，每张卡暴露稳定名称、选中状态和测试标识。
6. 静态预览上下文把内部操作降为不可交互视图，把内部输入设为只读、不可聚焦且 `tabIndex=-1`；只保留外层 22 张卡进入 Tab 顺序。
7. 静态预览使用样式形式的 `pointerEvents`，避免重新引入 React Native Web 的弃用警告。

### 8.2 可避免耗时与改进

1. **平台分文件造成页面单测盲区**
   - 现象：页面单测 mock 了通用输入组件，只证明父页面传参；Web 实际优先解析 `.web.tsx`，首轮没有转发新增属性。
   - 发现方式：独立代码审查对照模块解析路径发现，浏览器复验同步验证。
   - 改进：修改带 `.web`、`.native` 等平台文件的组件契约时，必须搜索全部同名实现，并至少增加一个目标平台组件级测试。
2. **动态固定预览使 DOM 数量多一份**
   - 现象：输入样式列表有 22 个预览，底部还渲染当前选中的第 23 个预览；首版 E2E 错按 22 计数。
   - 改进：先建立“列表实例 + 活动实例”的 DOM 模型，再断言总数；交叉审查应同时检查测试假设与生产实现。
3. **全页查询混入应用外壳**
   - 现象：首轮完整 E2E 使用全页 textbox 查询，除目标区域 23 个静态预览外，还计入应用外壳的 2 个隐藏输入，得到 25。
   - 浪费：一次 37/38 的完整 E2E 和一次重跑。
   - 改进：组件演示页的数量断言必须先限定到稳定页面容器；不要用全页计数推断局部组件结构。
4. **最终回归顺序**
   - 本批先完成定向 GREEN、Browser 复验和独立审查，再运行一次最终完整 E2E，避免在复审修订后重复长时间回归。

### 8.3 下批标准 SOP

1. 创建同级 worktree，并按锁文件独立安装依赖。
2. 本地临时关闭 Watchman，启动 Web；提交前逐字节恢复。
3. 初始化 Browser Control 账本，按页面分组执行“基线 → 动作 → 页面/Console/Network → 安全截图”。
4. 修改平台组件契约前先搜索 `.web`、`.native` 和默认实现；为实际目标平台增加组件级测试。
5. 先限定主内容范围，再点击稳定测试标识；已有登录态截图固定裁剪侧栏。
6. 完成 Browser 复验和独立审查后，只运行一次最终完整 E2E。
7. 串行执行完整 Vitest、Web export，避免与 CPU 密集型 E2E 并发。
8. 恢复主题、开发者模式、视口、临时配置和原页面，校验账本后再提交 PR。
